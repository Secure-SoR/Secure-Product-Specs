# Secure Platform — Release v5.1.0
**Release Date:** 18 February 2026  
**Baseline:** Secure Canonical Architecture v5  
**Scope:** 140 Aldersgate Corporate Occupier Account  

---

## Release Summary

Version 5.1.0 introduces a structural utility refactor based on billing source logic, routing corrections for Landlord Utilities, AI Workspace alignment with System-of-Record (SoR) principles, and stricter reporting consumption enforcement.

This release strengthens audit defensibility, lease-boundary clarity, and dataset traceability without introducing breaking changes to the canonical architecture.

## 1. Utilities Architecture Refactor

Utilities are now classified by:

- Billing Source  
- Control  
- Confidence  

Instead of commodity type (energy / water / heat).

### Final Utility Structure (140 Aldersgate)

#### Tenant Electricity (Direct – Measured)
- Billing Source: Tenant Direct  
- Control: Tenant Controlled  
- Confidence: Measured  
- Scope Classification: Scope 2 (Location-Based)  
- Floors: Ground, 4th, 5th  
- Supplier: Trio Submeters  

#### Landlord Utilities (Allocated)
- Billing Source: Landlord Recharge  
- Control: Landlord Controlled  
- Confidence: Allocated  
- Includes:
  - HVAC (MAPP recharge)
  - Water (provisional allocation)
  - Base-Building Electrical (common services)

#### Waste & Recycling (Third-Party)
- Billing Source: Third-Party  
- Control: Shared / Tenant Contract  
- Confidence: Measured  
- Contractor: Recorra  

### Removed

- Separate “Water (Landlord)” tile  
- Mixed tenant + landlord groupings  
- Standalone “Heat” category without physical meter

## 2. Routing Fix — Landlord Utilities (P0)

Landlord Utilities tile previously routed to:

`/data-library/energy`

and defaulted to Tenant Direct view.

Routing has been corrected so that Landlord Utilities now default to the Landlord Recharge dataset on load, either via query parameter handling or dedicated route.

## 4. Reporting SoR Enforcement

Reporting modules now strictly consume structured SoR datasets.

| Report Section | Source |
|---------------|--------|
| Organisational Boundary | `account.reportingBoundary` |
| Governance | Data Library → Governance & Accountability |
| Targets | Data Library → Targets & Commitments |
| Electricity | Tenant Direct records |
| Landlord Allocation | Landlord Utilities records |
| Waste | Third-Party records |

No embedded static governance or targets remain.

Reports remain pure consumption layers.

---

## 5. AI Agent Workspace Alignment

Route: `/ai-agents`

Workspace aligned to single-property context (140 Aldersgate).

### Canonical Agents (MVP)

- Data Readiness & Evidence Agent  
- Landlord–Tenant Boundary Agent  
- Action Prioritisation Agent  
- Reporting Copilot Agent  

### Agent Output Contract (Enforced)

Each agent modal must include:

1. Executive Summary  
2. Decision / Recommendation  
3. Evidence Links (Data Library references)  
4. Confidence Rating  
5. Data Gaps  
6. Next Actions (Owner + Due Date)  

---

## 6. Service Charge Handling Rule

If landlord does not provide a physical breakdown:

Create one consolidated record:

Landlord Utilities (Allocated)  
Includes: HVAC + Water  
Confidence: Allocated  

Do not create synthetic measured categories or duplicate allocations.

If a detailed breakdown is later provided, split into measured records.
